# Video Mode for Songs Implementation Plan

> **For Hermes:** Use subagent-driven-development to implement this plan task-by-task.

**Goal:** Add a toggle to switch between audio-only and video playback for songs that have YouTube videos.

**Architecture:** A `VideoPlayer` widget (using `media_kit`) replaces the album art in the now-playing screen when video mode is active. A `ValueNotifier<bool>` tracks the mode. The video stream URL is fetched from YouTube using the existing `ytid`. Video and audio are decoupled — when video mode activates, the existing audio keeps playing (muted) and the video renders alongside; the user can toggle back to audio-only at any time.

**Tech Stack:** `media_kit` (Windows video via libmpv), `youtube_explode_dart` (already present for YouTube API), Flutter's standard state management (`ValueNotifier`).

**Key constraint:** The Dart VM on this Windows dev machine cannot make outbound HTTP/HTTPS connections (SocketException 10057). All package downloads must use Python/curl workaround, and `package_config.json` may need manual updating. See skill `flutter-windows-port`.

### Task 2: Create `VideoPlayerWidget`

**Objective:** Build a reusable video player widget that renders YouTube video from a `ytid`.

**Files:**
- Create: `lib/widgets/video/youtube_video_player.dart`

**Step 1: Create player controller**

```dart
import 'package:flutter/material.dart';
import 'package:media_kit/media_kit.dart';
import 'package:media_kit_video/media_kit_video.dart';

class YoutubeVideoPlayer extends StatefulWidget {
  final String ytid;
  final bool playing;
  final VoidCallback? onVideoEnded;
  const YoutubeVideoPlayer({super.key, required this.ytid, this.playing = true, this.onVideoEnded});

  @override
  State<YoutubeVideoPlayer> createState() => _YoutubeVideoPlayerState();
}

class _YoutubeVideoPlayerState extends State<YoutubeVideoPlayer> {
  final Player player = Player();
  VideoController? videoController;
  bool _initialized = false;

  @override
  void initState() {
    super.initState();
    _initVideo();
  }

  Future<void> _initVideo() async {
    // Get the best video+audio stream URL from YouTube
    final manifest = await ytClient.videos.streamsClient.getManifest(widget.ytid);
    final videoStream = manifest.muxed?.withHighestBitrate() ?? manifest.videoOnly?.withHighestBitrate();
    if (videoStream == null) return;
    
    await player.open(Media(videoStream.url.toString()));
    if (widget.playing) player.play();
    videoController = VideoController(player);
    if (mounted) setState(() => _initialized = true);
  }

  @override
  void didUpdateWidget(YoutubeVideoPlayer old) {
    if (widget.playing != old.playing) {
      widget.playing ? player.play() : player.pause();
    }
    super.didUpdateWidget(old);
  }

  @override
  void dispose() {
    player.dispose();
    super.dispose();
  }

  @override
  Widget build(BuildContext context) {
    if (!_initialized || videoController == null) {
      return const Center(child: CircularProgressIndicator());
    }
    return ClipRRect(
      borderRadius: BorderRadius.circular(12),
      child: Video(
        controller: videoController!,
        fill: Colors.black,
      ),
    );
  }
}
```

**Step 2: Handle stream URL fetching**

The `ytClient` is already available globally. Create a helper function in a utility file (or add to `common_services.dart`):

```dart
Future<String?> getVideoStreamUrl(String ytid) async {
  try {
    final manifest = await ytClient.videos.streamsClient.getManifest(ytid);
    // Prefer muxed (video+audio) stream for direct playback
    final stream = manifest.muxed?.withHighestBitrate()
        ?? manifest.videoOnly?.withHighestBitrate();
    return stream?.url.toString();
  } catch (e) {
    return null;
  }
}
```

**Verification:** The widget renders a loading spinner initially, then a video when the stream resolves.

**Edge cases:**
1. **No video stream available** — show a "Video not available" message with the album art fallback
2. **Network error during video load** — retry once, then show a "Tap to retry" overlay

## Risks and Tradeoffs

1. **media_kit requires mpv** — On Windows, `media_kit` bundles mpv via a helper DLL. This adds ~30MB to the bundle and requires `mpv.exe` and `mpv-1.dll` to be distributed alongside `musify.exe`. The portable build script needs updating.
2. **Network restriction on this dev machine** — The Dart VM's SocketException (WSAENOTCONN 10057) prevents direct testing of video stream fetching. The feature will compile but won't play video until tested on a normal Windows machine.
3. **YouTube stream format changes** — Muxed (video+audio) streams from YouTube are less common now. Most streams are video-only + audio-only separate. media_kit can handle this with `MediaConfiguration`, but it adds complexity.
4. **Audio sync** — The audio plays via `just_audio` while video renders via `media_kit`. If the user toggles video ON mid-song, there will be a slight desync. Mitigation: mute the `just_audio` player and let `media_kit` handle both audio+video via muxed streams when possible.
